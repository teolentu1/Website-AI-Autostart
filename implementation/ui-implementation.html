SoundLab Game - UI Implementation

The graphical user interface is built using PySide6, Qt's official Python binding. The UI consists of two main pages: GameCreationPage (Game Engine) and PlayerPage (Game Player).

Threading Model

Two operations that are triggered from the GameCreationPage are too slow to run on the main thread.

  • AI Generation of a game (by the local LLM)
  • Saving of a game (text-to-speech generation for each node)

If executed on the main thread, the Qt event loop blocks and the UI freezes. To prevent this, both operations are executed in a background QThread.

Each of these operations is wrapped in a worker class: _GameGenerationWorker and _GameSaveWorker. The worker is moved to a QThread and started as seen below:

thread = QtCore.QThread(self)
       worker = _GameGenerationWorker(prompt)
       worker.moveToThread(thread)


       thread.started.connect(worker.run)
       worker.progress.connect(self._on_generation_progress)
       worker.finished.connect(self._on_generation_finished)
       worker.cancelled.connect(self._on_generation_cancelled)
       worker.failed.connect(self._on_generation_failed)
       worker.done.connect(self._on_generation_done)
       worker.done.connect(thread.quit)
       thread.finished.connect(worker.deleteLater)
       thread.finished.connect(self._on_generation_thread_finished)
       thread.finished.connect(thread.deleteLater)


       self._generation_thread = thread
       self._generation_worker = worker
       thread.start()

moveToThread transfers ownership of the worker object to the background thread. From that point, worker.run() executes entirely off the main thread. When it completes or fails, it emits a signal - finished, failed, or done, which Qt delivers back to the main thread where the connected slot updates the UI safely.

Signals for AI Generation

AI Generation is the most complex process handled by the game engine. It has multiple steps: blueprint generation and then node-by-node story generation, that produce real-time updates to the game graph in the GameCreationPage UI.

This is achieved using Qt signals that send updates to the UI during generation.

_GameCreationWorker defines 5 signals:

   progress = QtCore.Signal(dict)
   finished = QtCore.Signal(object)
   cancelled = QtCore.Signal(str)
   failed = QtCore.Signal(str)
   done = QtCore.Signal()

The worker passes self._emit_progress() to the game generator:

generator.generate_game(
               self.prompt,
               progress_cb=self._emit_progress,
               cancel_event=self._cancel_event,
           )

Whenever the pipeline produces intermediate results, it calls: progress_cb(payload). The workers _emit_progress method receives these payloads and emits them as a Qt signal.

@staticmethod
   def _emit_progress(progress_cb: Callable[[dict[str, Any]], None] | None, **payload: Any) -> None:
       if progress_cb is not None:
           progress_cb(payload)

Qt then delivers the signal to the main thread, where the UI updates safely.

This design ensures that:

  • The LLM pipeline remains UI-agnostic
  • Qt-specific logic is contained within the worker
  • Background work and UI updates remain cleanly separated
