System Architecture

The architecture is split into two independent applications sharing a common data layer.

The game creation engine is a PySide6 desktop application where a developer builds a branching story graph node by node. It optionally uses a local LLM to generate a full game from a text prompt, then saves the result as a .noui zip archive containing the serialised graph and all pre-generated TTS audio.

The game player is a separate PySide6 window that loads a .noui file and runs the game entirely through audio narration and hand gesture input.

LLM generation, TTS audio synthesis, and saving run on background QThreads to keep the UI responsive.

Design Patterns
Design Pattern Usage
Observer VideoGestureRecogniser is responsible for hand gesture recognition only. It has 2 subscribers, that are continuously updated about a hand gesture recognised, PowerManager and GestureController. PowerManager uses this data to manage power consumption of the VideoGestureRecogniser, while GestureController uses this data to startup/close applications.

Design Principles

  • Single Responsibility (SRP) — each component has one clear purpose, e.g. VideoGestureRecogniser, Talker
  • Don't Repeat Yourself (DRY) — code reuse is used heavily throughout the codebase
Design Pattern Usage
Observer This design pattern is also used to dynamically update the UI during the AI generation of the No-GUI game. After any progress has been made, classes from the game_generation_local_llm package update the UI, allowing the user to view the current progress of the game generation in real-time.
Strategy GamePlayer class is responsible for the main loop of the game. It interacts with the user using the dependency injected recogniser class. This class can be either VideoGestureRecogniser, or KeyboardInputHandler. Both of these classes have a public facing method get_gesture, simulating an interface in python. This allows the GamePlayer to use the recogniser without having to know whether the input is from the webcam or from the keyboard.
Mediator Game generation consists of two steps, generation of the blueprint using BlueprintGenerator, and then generation of the actual story using StoryGenerator. To avoid illogical dependencies, we have chosen to implement a third class GameGenerator, that mediates these two classes, calling one, and then the other. This class then reduces chaotic dependencies, enabling the objects to communicate only through the mediator GameGenerator class.

Data Storage

No-GUI Game

There is no database. All persistent data are stored in three places on the file system.

  • .noui files are the core data format. Each game is a zip file containing everything needed to play it: the graph structure and all audio. When the player loads a game, the archive is extracted to storageManager/temporary/ for the duration of the session.
  • settings.json lives at the project root and stores all player configuration:
    · input device (webcam or keyboard)
    · gesture bindings per action
    · keyboard key bindings
    · gesture recogniser timeout.
    It is created with defaults on first run and merged with defaults on every load so new fields are always present.
  • voiceSamples/ at the project root holds pre-generated .wav files for each of the 8 British Kokoro voices, used only for preview playback in the editor.

Inside a .noui archive the structure is:

GameName/
  graph.json
  audio/
    node_0_main.wav
    node_0_options.wav
    node_1_main.wav
    node_1_options.wav
    ...
    win.wav
    lose.wav
    quit.wav
    resume.wav
    start_new.wav
    progress_saved.wav
    main_menu_welcome.wav
    main_menu_new_game.wav
    main_menu_progress.wav
    inactivity.wav
  progress.json          ← only present if player quit mid-game

graph.json is a serialised SerialGraph - a flat dictionary of node IDs to SerialNode objects. Each SerialNode stores the scene text, left and right option labels, an adjacency list mapping option side codes ("0" = right, "1" = left) to target node IDs, and win/lose flags. Terminal nodes loop back to themselves in the adjacency list.

progress.json contains a single field of the node ID where the player last quit. It is written on quit, read on next load to offer resume/restart, and deleted when the player reaches a win or lose ending.
