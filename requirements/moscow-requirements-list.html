Functional Requirements

🟢 Must Have
  • The app should be running in background as a system tray application.
  • The app should monitor surroundings using the web cam while in use.
  • The app should recognise hand gestures done by the interacting user when in frame of the webcam.
  • The app should allow users to map hand gestures to applications of their choice.
  • The app should open applications on the user's Windows computer.
🔵 Should Have
  • The app should have a gesture reference page, showing the user what gestures it can recognise.
  • The app should allow users to remove entries of gesture mappings.
  • The app should open files (like .mp4) on the user's Windows computer.
🟡 Could Have
  • The app could work with the custom .noui file type to open NO-GUI games.
  • The app could have an option ‘requires webcam’ for applications that need access to webcam.
  • The app could hand off camera access to other applications if they are marked so.
  • The app could use a body model to isolate interacting users from the background.
  • The app could go to low power mode, upon user inactivity.
  • The app could provide a settings page to turn off/on the body model, and to define length of inactivity to turn on low power mode.
🔴 Won't Have
  • The app won't use Wifi Sensing to detect user presence.
  • The app won't allow use of different input devices other than a webcam (ie microphone).

Functional Requirements

🟢 Must Have
  • The game engine should have a 'Create new game' button, for creating a new NO-GUI game.
  • The game engine should provide a graph-like GUI for the game creation.
  • The GUI for the game graph should provide widgets for each node of the graph.
  • The GUI node widgets of the graph should each have a textbox for the main text, for the left option, and for the right option.
  • The GUI should allow a user to create new nodes for 'left' and 'right' options respectively.
  • The GUI should provide a textbox for the name of the game.
  • The GUI should have a save button that allows the user to save the game to a selected location.
  • The game engine should generate text-to-speech audio from the text provided by the user in the node widgets.
  • The game player should allow users to hear the audio of the game.
  • The game player should recognise hand gestures done by the user and use this as gameplay decisions.
  • The game generated by the game engine should be in the '.noui' filetype.
  • The '.noui' filetype should contain all the audio + graph structure, so that the game player can run the game.
🔵 Should Have
  • The game engine should have a button 'Edit game', for editing an already existing game.
  • The game engine GUI should allow a user to link already existing node widgets as left/right options of other nodes.
  • The game player should have a settings page, where the user can configure which hand gestures are used for which interaction.
  • The gameplay should start with a 'main menu' audio, saying the name of the game aloud.
  • The 'Welcome menu' of the gameplayer should have an option to start a new game, or load the previous game.
🟡 Could Have
  • The game engine could offer a variety of text-to-speech voices the user can choose from.
  • The game engine could use a LocalLLM to generate new graphs of games from story summaries.
  • The game engine could use a LocalLLM to edit already existing games.
  • The game player could have an option to use keyboard bindings instead of hand gestures to navigate the menu.
  • The game player could quit after user inactivity.
  • The game player settings page could allow the user to set the inactivity timeout.
🔴 Won't Have
  • The game engine should take in a pdf of a book and create a game from it.
  • The game player could have captions for transcription of the audio output.
  • The game engine could take in the user's custom audio, and base the text-to-speech audio output on that audio sample.

Non-Functional Requirements

🟢 Must Have
  • The app must function on Windows 11.
  • The app must provide a touch-free interaction.
  • The app must be compiled to an executable file or directory.
  • The app must run on Intel hardware.
  • The app must run all machine learning recognition offline.
🔵 Should Have
  • The app should be robust against random background gestures.
  • The app should respond to gestures within a reasonable time.
  • Gesture-to-application mappings should be stored locally and persist between sessions without data loss.
  • The app should be installable by a non-technical user without requiring manual configuration of dependencies.
🟡 Could Have
  • The app could run over long periods of time.
🔴 Won't Have
  • The app won't have multilingual support.

Non-Functional Requirements

🟢 Must Have
  • The NO-GUI engine should allow users to create NO-GUI games.
  • The NO-GUI player should allow the user to play NO-GUI games.
  • The NO-GUI player and engine should be compiled to an executable file or directory.
  • The NO-GUI player and engine should be 2 self-contained separate applications.
  • The applications must run on Windows 11.
🔵 Should Have
  • Low latency of gesture recognition while playing a NO-GUI game.
  • NO-GUI games should be easily distributable to users.
  • A NO-GUI game should be playable on any computer that has a NO-GUI game player installed.
🟡 Could Have
  • The NO-GUI player could start up within a reasonable time (e.g. under 5 seconds) on average consumer hardware.
  • The .noui file format could be versioned, so that older game files remain compatible with newer versions of the player.
🔴 Won't Have
  • The NO-GUI player won't enforce any digital rights management (DRM) on .noui game files.
  • The NO-GUI player won't support multiplayer or networked gameplay.
  • The NO-GUI engine won't support collaborative workspace.
  • The NO-GUI engine won't support partial saving.
