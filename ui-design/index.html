UI Design

Detailed overview of our UI design process for No-GUI Game and AI-Autostart

Design Principles

In designing AI-Autostart and the No-GUI Game, we prioritised several core Human-Computer Interaction (HCI) principles to create intuitive systems for our users.

Simplicity & Consistency: Design interfaces should be easy to understand and use by keeping them simple and maintaining a uniform user experience. All interfaces should be operable without extensive technical knowledge.
Tolerance: Systems should reduce the impact of user errors and allow easy recovery through undo, error prevention, and flexible input handling.
Visibility & Feedback: Every action and interactive element should have a clear and immediate response. Make important functions, controls, and system status visible on all interfaces.

User-centric requirements gathering

Our design process began with a user-centric approach by conducting an online survey with wheelchair users. We wanted to gain information about what difficulties users with accessibility needs face when approaching digital systems. The survey focused on their experiences with kiosk interfaces in high traffic environments.


Example response

What difficulties have you experienced when using kiosks?

Before accessibility buttons, it was really hard to reach things on the screen, so I always had to seek help. Most kiosks still don’t have them, but even when they do, it’s a hassle. Either I have to waste time looking for the button, or ask for assistance, who aren’t always the nicest.

Have you ever noticed an accessibility button that makes the interface more reachable?

I usually notice them after a few visits to the same place. They’re never that visible, and I only realize they exist once I’ve already struggled a few times.

How does needing to press an accessibility button make you feel?

  • ☐ I don't mind it
  • ☐ It's inconvenient but manageable
  • ☑ It makes me feel singled out

If kiosks could automatically adjust when you approached, how would that change your experience?

That would make a huge difference. It would feel like the system actually sees and respects me without me having to do anything special.

Wheelchair users reported:
  • Difficulty reaching elements on kiosk screens, often requiring assistance.
  • Accessibility buttons are usually hard to find and require time and effort to locate, leading to frustration and delays.
  • Many users only notice accessibility features after multiple visits, as they are not clearly visible in the interface.
  • The need to manually activate accessibility features can make users feel singled out and uncomfortable in public spaces.

This feedback highlighted the importance of visibility, simplicity, and inclusive design, leading us to explore different solutions for improving accessibility in interactions with digital interfaces.

This page documents the user interface design of the No-GUI Game project. The project has two distinct interfaces: the Game Engine, used by game creators to build and save branching story games, and the Game Player, used to load and play them.

Since the game play experience is entirely audio and gesture based, the Game Player GUI is as minimal as possible.

User-Centric Requirements Gathering

We began our design process by analysing the nature and needs of our users. We identified our two main user groups as the blind and visually impaired, and autistic children.

User Group 1: Blind and Visually Impaired

In our research for this user group we came up with a persona: a blind 14–17 year old boy Kyle Bell. We conducted a mock interview and the responses made the problems very clear:

What parts of the setup process are most frustrating or time-consuming for you?

"I have to wait until there's someone around who can help, and that's not always the case."

Do you think this affects your social life or how connected you feel to others?

"I end up missing out — not because I don't want to play, but because it's not accessible."

Derived requirements:
  • The player must be able to initiate a game entirely independently.
  • Every interaction during play must be communicated through audio.
  • The setup process must be short and require as few steps as possible.
User Group 2: Autistic Children and their Teachers

Our second user group is autistic children, particularly in school settings where teachers want interactive scenarios for social skills training. Research into autism education highlighted that many autistic children benefit from predictable, repeatable scenarios where they can practise social situations in a low-pressure, low-stimulus environment [1].

Derived requirements:
  • A non-technical user must be able to create a complete game without writing code.
  • Generated audio must be consistent and predictable across repeated plays.
  • The graph structure must be simple enough that a teacher can understand, edit and verify generated games before using them with the children.
  • The No-GUI games should be relatively low-stimulus.
Initial Design

We also made some sketches to plan the page layouts for the Game Engine: focusing on how the AI panel should sit alongside the canvas without cluttering it and how all the menus and dropdowns should be placed. For the AI panel, we took inspiration from how Github Copilot is integrated into VS Code — a simple side panel [2].

Prototyping & Iterations
Game Engine — Canvas and Graph Layout
  • Fixed top-down layout: Initially the game graph had a fixed top-down layout where nodes could not be moved. Child nodes were added to the bottom of the graph with a set distance between them. This led to overlap as the tree grew and also meant that the game was limited to a binary tree causing exponential growth.
  • BFS-based automatic layout: The tree layout algorithm was then replaced with a BFS-based automatic layout that places nodes at depths corresponding to their distance from the root, which makes the branching structure immediately readable without any manual arrangement by the creator.
  • Zoom and Scroll: The zoomable and scrollable canvas was added after it became clear that games with more than 6–8 nodes did not fit on screen at the default zoom level.
  • Voice Preview: Later on in development we added the option to generate games using one of 8 audio narrators offered by Kokoro TTS. This was integrated into the UI as a dropdown next to the title bar. We also added a Preview button that plays a pre-recorded audio file narrated by the selected voice which lets the user test out voices before generating the game.
Node Widget Design
  • The initial node was simple: three text fields (the scene text and two option labels) and two add-child buttons for left and right options.
  • Win and lose controls: We then added the option to set as win or loss after starting to test out the game play experience.
  • Delete button: A delete button was added to leaf nodes after it became clear that removing incorrect or unwanted nodes was essential to a usable editing workflow. Only leaf nodes can be deleted to prevent accidentally disconnecting subtrees.
  • Link mode: We identified a need for connections that skip directly to an existing node rather than always creating a new child. A Link button was added to each node: pressing it enters linking mode, in which all eligible target nodes on the canvas are highlighted. Clicking a highlighted node creates the connection and exits linking mode. This allows non-binary graph structures where paths converge.
AI Generation
  • The first version generated the entire game in one LLM call and only showed the result after generation was complete, which meant the creator waited several minutes with no feedback.
  • Live generation of nodes: The second iteration, the current version, displays the graph layout with placeholder nodes while generation is running, slowly filling in the nodes.
Game Player
  • No Settings: Initially we did not think about adding a settings page where the user would be able to set gestures, but as we experimented with user testing we realized our users may have limitations on what gestures they can comfortably perform.
  • Gesture Binding: A settings dialog was added that let the user change gesture to action bindings from a dropdown.
  • Keyboard Mode: We added a keyboard mode for sighted users who prefer key presses over gestures. This required adding an input mode toggle and key-to-action bindings section to the settings dialog.
  • Inactivity Timeout: Added an inactivity timeout function to the game player that auto-saves progress and exits after a set period of time passes without a gesture being recognized. This was made customizable to accommodate differing reaction times of users — the waiting period can be set from 0 mins 0 seconds to 59 mins 59 seconds.
Heuristic Evaluations & Improvements
Heuristic Principle Problem Solution
Visibility of System Status The original editor gives no feedback during saving, which could take several minutes due to TTS generation. Added modal progress dialog so the creator knows the save is in progress and the UI does not appear frozen.
User Control and Freedom The AI generation panel had no way to stop generation once started. Added Stop button that cancels the LLM pipeline and leaves whatever nodes had already been generated on the canvas for the creator to work with.
Error Prevention The settings dialog originally allowed the user to assign the same key or gesture to multiple actions, which would silently break input. Added duplicate validation on save, with a clear warning message identifying the conflict.
Consistency and Standards The arrows linking nodes were all one color making it difficult to differentiate between left and right options. Used a red line for linking all right options and a blue line for all left.
Final Design

In the end our UI design went through many changes as the project grew and received feedback from testers. Our final version of our interfaces are detailed below.

Both interfaces support dark mode, applied automatically based on the user's operating system appearance settings — no manual toggle is required.

Game Player

The game player interface is deliberately minimal. Its primary user may be visually impaired or have their eyes occupied, so the UI exists only to get the player into the game — everything after that is audio.

  • Player launcher — a small window with a file path field, a Browse button to select a .noui file, a Run button, and a Settings button. Once a file is selected, the user clicks Run. The game loop starts immediately on a background thread and begins narrating the main menu audio. The UI does not need to be looked at again.
  • Settings dialog — accessible before starting a game. Gives the player control over: the recogniser being used (keyboard or video), gesture bindings for video recognition, key bindings for keyboard mode, and the inactivity timeout — the period after which the game auto-saves progress and exits.
Game Engine

The game creation engine is a PySide6 desktop application built for game creators.

  • Home page — the application opens to a minimal home screen with two options: Create a New Game and Edit Game. The home page is intentionally sparse, with no unnecessary navigation menus or settings visible.
  • Game creation page — the main editor view is divided into two sections. The left-hand side is a zoomable, scrollable node canvas displaying the game's branching story graph. Each node contains the scene text, a left option label, a right option label, and controls to set it as a win or lose node. Blue lines connect left option paths and red lines connect right option paths. The canvas can be zoomed with the scroll wheel and panned by dragging.
  • Title bar — contains the game title field, a voice selector dropdown (listing all 8 British Kokoro voices by name and gender), and a Preview button that plays a short audio sample of the selected voice from a pre-generated file.
  • AI generation panel — the right-hand panel is the AI game generator. A text area at the bottom lets the creator describe their game, which is sent as a prompt to the local LLM. The canvas updates in real time as each node is generated, showing placeholder text for nodes not yet complete. A progress bar and status label track the generation state. A Stop button allows the creator to cancel generation at any time. Once generation is complete, the creator can edit any node manually before saving.
